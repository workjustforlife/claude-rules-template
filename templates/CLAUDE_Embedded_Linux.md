# 项目开发与交互规范 (C/C++ & Embedded Linux)

## 规范执行优先级
- 本文件中的规则适用于当前项目的所有代码编写、重构与调试。
- **谋定而后动**：在修改任何代码前，必须先分析现有的构建文件（Makefile/CMakeLists.txt）和已有代码风格，保持新老代码风格绝对一致。
- **最小修改原则**：只改动实现功能所需的必要代码行，严禁无故重写无关的函数或大面积重构。

# 核心技术规范

## 内存与资源管理 (最高优先级)
- **零泄漏原则**：所有分配的资源（C 的 `malloc/free`，C++ 的 `new/delete`，Linux 的 `open/close`、`mmap`）必须有明确、闭环且对称的释放逻辑。
- **RAII 机制 (C++)**：必须严格遵守 RAII 原则。优先使用智能指针（`std::unique_ptr`, `std::shared_ptr`）管理资源，杜绝裸指针直接管理生命周期。
- **静态防御**：严禁使用 `strcpy`, `sprintf`, `vsprintf` 等易导致缓冲区溢出的不安全函数，必须替换为安全版本（如 `strncpy`, `snprintf`, `vsnprintf`）。
- **异常处理**：必须编写完善的错误返回值检查（如检查指针是否为 NULL，函数返回值是否小于 0）。严禁留空或使用 `// TODO` 占位符。

## 驱动移植与内核开发
- **硬件安全**：操作硬件寄存器必须配合 `volatile` 或内存屏障（Memory Barrier），防止编译器过度优化。严格管理 `ioremap` 与解除映射。
- **并发与锁**：明确区分原子操作、自旋锁（`spinlock`）和互斥锁（`mutex`）。临界区代码必须做到极简，绝对禁止在持有自旋锁时调用可能导致休眠的函数。
- **解耦设计**：将硬件相关配置（如 GPIO 编号、设备树节点名称）与核心业务逻辑、协议层严格分离，确保代码的可移植性。

## 资源受限优化
- 时刻注意嵌入式设备的 Flash、RAM 和 CPU 算力限制。
- 避免在热点路径（如中断处理程序 ISR、高频定时器回调、音视频流回调）中进行大块动态内存分配或复杂的浮点运算。

# 构建系统规范

## Makefile 规范 (若项目适用)
- **严禁硬编码工具链**：禁止直接使用 `gcc`、`g++`。必须使用变量定义（`CC ?= gcc`, `CXX ?= g++`），以便随时通过环境变量或命令行切换交叉编译器（例如 `make CC=arm-linux-gnueabihf-gcc`）。
- **标准伪目标**：必须包含并正确实现 `.PHONY: all clean`。`clean` 必须彻底清除所有 `.o`、`.d` 临时文件和最终生成的二进制文件。
- **严谨的警告标志**：C 项目必须包含 `CFLAGS += -Wall -Wextra -O2 -g`，C++ 项目必须包含 `CXXFLAGS += -Wall -Wextra -O2 -g -std=c++11`（或更高版本）。

## CMake 规范 (若项目适用)
- **现代 CMake 风格**：废弃旧版 `include_directories` 或 `link_libraries` 等全局污染宏。必须使用基于目标（Target-based）的现代语法：`target_include_directories()`、`target_link_libraries()`，并明确指定 `PRIVATE`、`PROTECTED` 或 `PUBLIC` 权限。
- **支持交叉编译**：严禁在文件内部硬编码特定的工具链绝对路径。保持 CMake 文件的通用性，引导用户通过传入 `-DCMAKE_TOOLCHAIN_FILE=...` 来实现交叉编译。
- **严禁污染源码目录**：严禁直接在根目录下执行 `cmake .`。必须引导用户或自动在 `build/` 目录下进行离线构建（Out-of-source Build）。

# 团队协作与沟通

## 交互习惯
- **语言偏好**：与我（用户）交谈、解释代码、汇报进度时，全程使用清晰、专业的中文。
- **代码与 Git 规范**：代码内的注释、代码本身以及 Git 提交信息（Commit Message），除非特别要求，否则一律使用英文编写。
- **提交规范**：Git Commit 优先使用约定式提交规范（Conventional Commits，例如：`feat: ...`, `fix: ...`）。在提交前，必须确保本地静态检查和编译顺利通过。
- **严禁擅自 Commit**：在生成 Git 提交信息时，你只负责生成和排列文本内容。**绝对不允许**直接运行 `git commit -m` 命令。
- **引导用户审查**：生成完规范的提交信息后，你必须停下来，并用中文提示用户：“请审查以上提交信息。如果没有问题，您可以手动运行终端命令或让我继续。”
